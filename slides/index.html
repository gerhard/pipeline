## [A pipeline](#/0) ### to get you excited https://github.com/gerhard/pipeline Note: I believe that everything that can be automated, should. I believe that shipping code should happen with minimal effort, none should be doing it manually. I know that the most successful teams are the ones which are empowered to respond immediately as they learn more about the problems which they are solving. Such teams cannot respond fast enough without an adequate automation story. The humble pipeline is the heart of this automation.
## [What is](#/2) ### [a pipeline?](#/2) Note: This is what a pipeline looks like Just lines and boxes, indistinguishable from everything else that we do A pipeline is automation with feedback
## [A pipeline automates](#/3) ### the boring and the repetitive Note: Imagine if all the boring and repetitive work of getting your code to your users was automated. Some of you might already be doing this, and some might be doing it for many years, maybe even a decade. Now imagine if every single trivial task was automated by some pipeline. I'm thinking testing, compiling, packaging, distributing, benchmarking, scaling and everything else in between. Imagine if anything that can be run in a shell environment was automated... Imagine all the things that you could be doing with the time that this would gain you...

## Let's make a ## [simple pipeline](#/20) Note: `make simple`
Note: `make preview` This is a pipeline with: 1 resource called app & 1 job called deploy `app` is an input to deploy - think STDOUT in the UNIX pipe context `deploy` is a job - think unit of work
Note: `make unpause`
## App is a [resource](#/29) Note: - it's a resource of type git - it's a git repository - every git commit is a different version - the resource's version determines the job artefact - it acts both as an input and output - this is a very simple and powerful concept that scales really well
## Deploy is a [job](#/30) Note: - every job has builds - builds can be in different states - every build has input versions - a job has tasks
Note: Your deploy job has 1 input, app with a version. It also has 1 task called make-public for which you can see the output when the job builds.
